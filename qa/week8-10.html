Introduction to Test-First Approaches

Test-first approaches involve writing tests before writing the code that will be tested. This reversal of the traditional development process has significant benefits for code quality, design, and maintainability.

The Cost of Change Curve

One of the key motivations for test-first approaches is addressing the "cost of change curve" - the observation that the cost of fixing defects increases exponentially the later they are found in the development lifecycle.

Benefits of Test-First Approaches

  • Early Defect Detection - Find issues before they become expensive to fix
  • Better Design - Writing tests first encourages more modular, testable code
  • Clear Requirements - Tests serve as executable specifications
  • Confidence in Changes - Tests provide a safety net for refactoring
  • Documentation - Tests document how the code is supposed to work
  • Focus - Developers concentrate on meeting specific requirements

Test-Driven Development (TDD)

Test-Driven Development is a software development process where tests are written before the code that will be tested. It follows a simple cycle: write a failing test, write the minimum code to make the test pass, then refactor.

The TDD Cycle

Interactive Diagram: The TDD Cycle

Red
1. Write a Failing Test

Start by writing a test that defines a function or improvement of a function, which should fail because the function isn't implemented yet.

This step clarifies what you're trying to achieve before you write any code.

→
Green
2. Make the Test Pass

Write the minimum amount of code necessary to make the test pass.

Don't worry about code quality yet - focus on making it work.

→
Refactor
3. Improve the Code

Clean up the code while keeping the tests passing.

Remove duplication, improve names, and ensure the code follows good design principles.

→

TDD Example

Example: TDD Process for a Simple Calculator
// Step 1: Write a failing test (Red)
// calculator.test.js
const Calculator = require('./calculator');

describe('Calculator', () => {
  test('should add two numbers correctly', () => {
    const calculator = new Calculator();
    expect(calculator.add(2, 3)).toBe(5);
  });
});

// Running this test will fail because Calculator doesn't exist yet

// Step 2: Write the minimum code to make the test pass (Green)
// calculator.js
class Calculator {
  add(a, b) {
    return 5; // Hardcoded return value just to make the test pass
  }
}

module.exports = Calculator;

// The test now passes, but the implementation is obviously not correct

// Step 3: Refactor the code while keeping the test passing
// calculator.js
class Calculator {
  add(a, b) {
    return a + b; // Proper implementation
  }
}

module.exports = Calculator;

// The test still passes with the improved implementation

// Step 4: Add another test (Red again)
// calculator.test.js
describe('Calculator', () => {
  test('should add two numbers correctly', () => {
    const calculator = new Calculator();
    expect(calculator.add(2, 3)).toBe(5);
  });
  
  test('should subtract two numbers correctly', () => {
    const calculator = new Calculator();
    expect(calculator.subtract(5, 2)).toBe(3);
  });
});

// This test will fail because subtract() doesn't exist yet

// Step 5: Write the minimum code to make the new test pass (Green)
// calculator.js
class Calculator {
  add(a, b) {
    return a + b;
  }
  
  subtract(a, b) {
    return a - b;
  }
}

module.exports = Calculator;

// Both tests now pass

// Continue the cycle for multiply, divide, etc.

TDD Best Practices

  • Keep Tests Small - Test one concept per test
  • Run Tests Frequently - Run the tests after every small change
  • Write the Simplest Code - Don't implement more than needed to pass the test
  • Refactor Regularly - Clean up code as soon as tests pass
  • Test Behavior, Not Implementation - Focus on what the code does, not how it does it
  • Use Descriptive Test Names - Make it clear what each test is checking

Exercise: TDD Practice

Using TDD, implement a function that checks if a string is a palindrome (reads the same backward as forward). Follow these steps:

  1. Write a test for the simplest case (e.g., "a" is a palindrome)
  2. Write the minimum code to make the test pass
  3. Write a test for a more complex case (e.g., "racecar" is a palindrome)
  4. Update the code to make both tests pass
  5. Write a test for a non-palindrome (e.g., "hello" is not a palindrome)
  6. Update the code to make all tests pass
  7. Write a test for a palindrome with spaces and different cases (e.g., "A man a plan a canal Panama")
  8. Update the code to handle this case
  9. Refactor your code while keeping all tests passing

Behavior-Driven Development (BDD)

Behavior-Driven Development extends TDD by focusing on the behavior of the application from the user's perspective. It uses natural language constructs to express tests in a way that non-technical stakeholders can understand.

Key Concepts of BDD

  • Ubiquitous Language - Using the same terminology across all stakeholders
  • Outside-In Development - Starting with user-facing behavior and working inward
  • Collaboration - Involving business stakeholders in defining expected behavior
  • Scenarios - Describing features using Given-When-Then format
  • Executable Specifications - Turning scenarios into automated tests

Given-When-Then Format

BDD scenarios typically follow the Given-When-Then format:

  • Given - The initial context or preconditions
  • When - The event or action that occurs
  • Then - The expected outcome or result

Example: BDD Scenario for User Login

Feature: User Login
  As a registered user
  I want to log in to the application
  So that I can access my account

  Scenario: Successful login with valid credentials
    Given I am on the login page
    When I enter valid username and password
    And I click the login button
    Then I should be redirected to the dashboard
    And I should see a welcome message with my name

  Scenario: Failed login with invalid password
    Given I am on the login page
    When I enter a valid username
    And I enter an invalid password
    And I click the login button
    Then I should see an error message
    And I should remain on the login page

BDD Frameworks

Several frameworks support BDD-style testing:

  • Cucumber - Supports Gherkin syntax for defining scenarios
  • Jest with Cucumber - Combines Jest testing with Cucumber for JavaScript
  • Jasmine - BDD framework for JavaScript
  • Mocha with Chai - Flexible JavaScript testing with BDD-style assertions
  • SpecFlow - BDD framework for .NET
Example: BDD with Cucumber.js
// login.feature
Feature: User Login
  As a registered user
  I want to log in to the application
  So that I can access my account

  Scenario: Successful login with valid credentials
    Given I am on the login page
    When I enter valid username and password
    And I click the login button
    Then I should be redirected to the dashboard
    And I should see a welcome message with my name
// login-steps.js
const { Given, When, Then } = require('@cucumber/cucumber');
const { Builder, By, until } = require('selenium-webdriver');
const assert = require('assert');

let driver;

Given('I am on the login page', async function() {
  driver = await new Builder().forBrowser('chrome').build();
  await driver.get('https://example.com/login');
});

When('I enter valid username and password', async function() {
  await driver.findElement(By.id('username')).sendKeys('testuser');
  await driver.findElement(By.id('password')).sendKeys('password123');
});

When('I click the login button', async function() {
  await driver.findElement(By.id('login-button')).click();
});

Then('I should be redirected to the dashboard', async function() {
  await driver.wait(until.urlContains('/dashboard'), 5000);
  const currentUrl = await driver.getCurrentUrl();
  assert(currentUrl.includes('/dashboard'), 'URL should contain /dashboard');
});

Then('I should see a welcome message with my name', async function() {
  const welcomeMessage = await driver.findElement(By.id('welcome-message')).getText();
  assert(welcomeMessage.includes('testuser'), 'Welcome message should include username');
  
  // Clean up
  await driver.quit();
});

BDD vs. TDD

Aspect TDD BDD
Focus Implementation details User behavior and business value
Language Technical, code-oriented Natural language, business-oriented
Primary Audience Developers All stakeholders
Test Scope Usually unit tests Often acceptance or integration tests
Starting Point Individual functions or methods User stories or features
Development Approach Inside-out Outside-in

Continuous Integration with Testing

Continuous Integration (CI) is a development practice where developers integrate code into a shared repository frequently, and each integration is verified by an automated build and automated tests.

CI Process

→
Test

Automated tests are run (unit, integration, etc.)

Code quality checks are performed

→

Results are reported to the team

Notifications are sent for build failures

→
Deploy

If all tests pass, code can be deployed to staging

In CI/CD, this may continue to production

CI Tools

  • Jenkins - Open-source automation server
  • GitHub Actions - CI/CD service integrated with GitHub
  • CircleCI - Cloud-based CI/CD platform
  • Travis CI - CI service for open-source projects
  • GitLab CI - CI/CD integrated with GitLab
  • Azure DevOps - Microsoft's DevOps service
Example: GitHub Actions Workflow for JavaScript Testing
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        node-version: [14.x, 16.x, 18.x]
    
    steps:
    - uses: actions/checkout@v2
    
    - name: Use Node.js ${{ matrix.node-version }}
      uses: actions/setup-node@v2
      with:
        node-version: ${{ matrix.node-version }}
        cache: 'npm'
    
    - name: Install dependencies
      run: npm ci
    
    - name: Lint
      run: npm run lint
    
    - name: Run unit tests
      run: npm test
    
    - name: Run integration tests
      run: npm run test:integration
    
    - name: Upload coverage reports
      uses: codecov/codecov-action@v1
      with:
        token: ${{ secrets.CODECOV_TOKEN }}
        file: ./coverage/lcov.info
        fail_ci_if_error: true

Benefits of CI with Testing

  • Early Detection of Issues - Problems are found as soon as they're introduced
  • Reduced Integration Problems - Frequent integration minimizes conflicts
  • Faster Feedback - Developers know quickly if their changes work
  • Improved Code Quality - Automated tests and quality checks enforce standards
  • Increased Confidence - Team knows the code is always in a working state
  • Reduced Manual Testing - Automation handles repetitive testing tasks

Exercise: Setting Up a CI Pipeline

Create a simple CI pipeline for a JavaScript project using GitHub Actions:

  1. Create a new GitHub repository
  2. Initialize a simple Node.js project with Jest for testing
  3. Write a simple function and a test for it
  4. Create a GitHub Actions workflow file (.github/workflows/test.yml)
  5. Configure the workflow to run tests on push and pull requests
  6. Push your code to GitHub and verify that the workflow runs
  7. Intentionally break a test and push again to see the workflow fail
  8. Fix the test and push again to see the workflow succeed

Integrating Testing into Agile Methods

Agile development methods emphasize iterative development, collaboration, and responding to change. Testing is an integral part of Agile processes, not a separate phase.

Testing in Scrum

  • Definition of Done - Includes passing all tests
  • Sprint Planning - Testing tasks are included in sprint backlog
  • Daily Scrum - Testing progress is discussed alongside development
  • Sprint Review - Demonstrates working, tested features
  • Sprint Retrospective - Reflects on testing practices and improvements

Testing in Extreme Programming (XP)

  • Test-Driven Development - Core practice of XP
  • Pair Programming - Developers and testers work together
  • Continuous Integration - Tests run automatically with each integration
  • Small Releases - Frequent delivery of tested features
  • Collective Ownership - Everyone is responsible for test quality

Continuous Testing

Continuous Testing is the process of executing automated tests as part of the software delivery pipeline to obtain immediate feedback on the business risks associated with a software release candidate.

  • Shift Left - Moving testing earlier in the development process
  • Shift Right - Extending testing into production environments
  • Test Automation - Automating tests at all levels
  • Feedback Loops - Providing rapid feedback to developers
  • Risk-Based Testing - Focusing on areas with the highest risk

Real-World Example: Spotify's Testing Approach

Spotify has developed a unique approach to testing that balances speed and quality in their agile development process.

Key Aspects of Spotify's Approach

  • Squad Autonomy - Each team (squad) decides its own testing strategy
  • Testing Guilds - Cross-team groups that share testing knowledge
  • Test Automation - Heavy emphasis on automated testing
  • Feature Toggles - Using feature flags to control rollout and testing
  • A/B Testing - Testing new features with a subset of users
  • Monitoring - Using production monitoring as a form of testing

Lessons from Spotify

  • Balance autonomy with alignment
  • Invest in test automation infrastructure
  • Use feature toggles to reduce risk
  • Learn from real user behavior
  • Share knowledge across teams
