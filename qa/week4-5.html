Weeks 4-5: Test Design Concepts

Learning Objectives

By the end of these two weeks, you will be able to:

  • Understand the fundamental concepts of test design
  • Apply different test design techniques to create effective test cases
  • Identify and implement appropriate coverage criteria for testing
  • Design tests based on specifications, structure, and experience
  • Create test cases that maximize defect detection with minimal effort
  • Evaluate the effectiveness of test designs using coverage metrics

Introduction to Test Design

Test design is the process of creating and developing test cases based on specific test conditions. Effective test design is crucial for finding defects efficiently and providing confidence in the quality of the software.

The Test Design Process

  1. Analyze the Test Basis - Understand requirements, specifications, and other documentation
  2. Identify Test Conditions - Determine what aspects need to be tested
  3. Design Test Cases - Create specific test scenarios with inputs, actions, and expected results
  4. Define Test Data - Specify the data needed to execute the tests
  5. Define Test Environment - Specify the environment requirements for testing
  6. Create Traceability - Link test cases to requirements and other artifacts

Test Design Techniques Overview

Test design techniques help testers create effective test cases systematically. They can be categorized into three main types:

  • Specification-Based (Black-Box) Techniques - Based on the external behavior of the software
  • Structure-Based (White-Box) Techniques - Based on the internal structure of the software
  • Experience-Based Techniques - Based on the tester's knowledge and experience

Discussion Question

Why is it important to use a combination of different test design techniques rather than relying on just one approach? What are the potential risks of using only one type of technique?

Specification-Based (Black-Box) Techniques

Specification-based techniques derive tests from the functional specifications without knowledge of the internal structure of the component or system.

Equivalence Partitioning

Equivalence partitioning divides input data into groups (partitions) that are expected to exhibit similar behavior, allowing you to test one value from each partition.

Example: Age Validation

For a field that accepts ages between 18 and 65:

  • Invalid Partition (below minimum): Test with age 17
  • Valid Partition: Test with age 30
  • Invalid Partition (above maximum): Test with age 66
Example: Implementing Equivalence Partitioning Tests in JavaScript
// Function to validate age for a job application
function validateAge(age) {
  if (typeof age !== 'number') {
    return 'Invalid input: Age must be a number';
  }
  
  if (age < 18) {
    return 'Invalid age: Applicant must be at least 18 years old';
  }
  
  if (age > 65) {
    return 'Invalid age: Applicant must be 65 years or younger';
  }
  
  return 'Valid age';
}

// Test cases using equivalence partitioning
describe('Age Validation Tests', () => {
  // Invalid type partition
  test('Invalid input type', () => {
    expect(validateAge('twenty')).toBe('Invalid input: Age must be a number');
  });
  
  // Invalid partition (below minimum)
  test('Age below minimum (17)', () => {
    expect(validateAge(17)).toBe('Invalid age: Applicant must be at least 18 years old');
  });
  
  // Valid partition
  test('Valid age (30)', () => {
    expect(validateAge(30)).toBe('Valid age');
  });
  
  // Invalid partition (above maximum)
  test('Age above maximum (66)', () => {
    expect(validateAge(66)).toBe('Invalid age: Applicant must be 65 years or younger');
  });
});

Boundary Value Analysis

Boundary value analysis focuses on testing at and around the boundaries of input partitions, as defects often occur at the boundaries.

Example: Age Validation Boundaries

For a field that accepts ages between 18 and 65:

  • Lower Boundary: Test with ages 17, 18, 19
  • Upper Boundary: Test with ages 64, 65, 66
Example: Implementing Boundary Value Analysis Tests in JavaScript
// Using the same validateAge function from before

// Test cases using boundary value analysis
describe('Age Validation Boundary Tests', () => {
  // Lower boundary tests
  test('Age just below lower boundary (17)', () => {
    expect(validateAge(17)).toBe('Invalid age: Applicant must be at least 18 years old');
  });
  
  test('Age at lower boundary (18)', () => {
    expect(validateAge(18)).toBe('Valid age');
  });
  
  test('Age just above lower boundary (19)', () => {
    expect(validateAge(19)).toBe('Valid age');
  });
  
  // Upper boundary tests
  test('Age just below upper boundary (64)', () => {
    expect(validateAge(64)).toBe('Valid age');
  });
  
  test('Age at upper boundary (65)', () => {
    expect(validateAge(65)).toBe('Valid age');
  });
  
  test('Age just above upper boundary (66)', () => {
    expect(validateAge(66)).toBe('Invalid age: Applicant must be 65 years or younger');
  });
});

Decision Table Testing

Decision table testing is used when different combinations of inputs result in different actions. It's particularly useful for complex business rules.

Example: Discount Calculation

Conditions Actions
Rule Member? Purchase > $100? Holiday Sale? Discount %
1 Yes Yes Yes 20%
2 Yes Yes No 15%
3 Yes No Yes 15%
4 Yes No No 10%
5 No Yes Yes 10%
6 No Yes No 5%
7 No No Yes 5%
8 No No No 0%
Example: Implementing Decision Table Tests in JavaScript
// Function to calculate discount
function calculateDiscount(isMember, purchaseAmount, isHolidaySale) {
  if (isMember) {
    if (purchaseAmount > 100) {
      return isHolidaySale ? 20 : 15;
    } else {
      return isHolidaySale ? 15 : 10;
    }
  } else {
    if (purchaseAmount > 100) {
      return isHolidaySale ? 10 : 5;
    } else {
      return isHolidaySale ? 5 : 0;
    }
  }
}

// Test cases using decision table testing
describe('Discount Calculation Tests', () => {
  // Rule 1: Member, Purchase > $100, Holiday Sale
  test('Member with purchase > $100 during holiday sale gets 20% discount', () => {
    expect(calculateDiscount(true, 150, true)).toBe(20);
  });
  
  // Rule 2: Member, Purchase > $100, No Holiday Sale
  test('Member with purchase > $100 outside holiday sale gets 15% discount', () => {
    expect(calculateDiscount(true, 150, false)).toBe(15);
  });
  
  // Rule 3: Member, Purchase <= $100, Holiday Sale
  test('Member with purchase <= $100 during holiday sale gets 15% discount', () => {
    expect(calculateDiscount(true, 75, true)).toBe(15);
  });
  
  // Rule 4: Member, Purchase <= $100, No Holiday Sale
  test('Member with purchase <= $100 outside holiday sale gets 10% discount', () => {
    expect(calculateDiscount(true, 75, false)).toBe(10);
  });
  
  // Rule 5: Non-Member, Purchase > $100, Holiday Sale
  test('Non-member with purchase > $100 during holiday sale gets 10% discount', () => {
    expect(calculateDiscount(false, 150, true)).toBe(10);
  });
  
  // Rule 6: Non-Member, Purchase > $100, No Holiday Sale
  test('Non-member with purchase > $100 outside holiday sale gets 5% discount', () => {
    expect(calculateDiscount(false, 150, false)).toBe(5);
  });
  
  // Rule 7: Non-Member, Purchase <= $100, Holiday Sale
  test('Non-member with purchase <= $100 during holiday sale gets 5% discount', () => {
    expect(calculateDiscount(false, 75, true)).toBe(5);
  });
  
  // Rule 8: Non-Member, Purchase <= $100, No Holiday Sale
  test('Non-member with purchase <= $100 outside holiday sale gets 0% discount', () => {
    expect(calculateDiscount(false, 75, false)).toBe(0);
  });
});

Use Case Testing

Use case testing focuses on the user's interaction with the system to achieve specific goals.

Example: User Registration Use Case

Basic Flow:

  1. User navigates to registration page
  2. User enters valid information
  3. System validates information
  4. System creates account
  5. System sends confirmation email
  6. User is redirected to login page

Alternative Flows:

  • User enters password that doesn't meet requirements

Structure-Based (White-Box) Techniques

Structure-based techniques derive tests based on the internal structure of the component or system.

Statement Coverage

Statement coverage aims to execute each statement in the code at least once.

Example: Statement Coverage
function processOrder(order) {
  let total = 0;
  
  // Statement 1
  for (const item of order.items) {
    // Statement 2
    total += item.price * item.quantity;
  }
  
  // Statement 3
  if (total > 100) {
    // Statement 4
    total *= 0.9; // 10% discount
  }
  
  // Statement 5
  return total;
}

// Test for statement coverage
describe('Process Order - Statement Coverage', () => {
  test('Order with total <= 100', () => {
    const order = {
      items: [
        { name: 'Item 1', price: 20, quantity: 2 }
      ]
    };
    
    expect(processOrder(order)).toBe(40);
    // This test covers statements 1, 2, 3, 5
  });
  
  test('Order with total > 100', () => {
    const order = {
      items: [
        { name: 'Item 1', price: 60, quantity: 2 }
      ]
    };
    
    expect(processOrder(order)).toBe(108); // 120 - 10% = 108
    // This test covers statements 1, 2, 3, 4, 5
  });
  
  // Together, these tests achieve 100% statement coverage
});

Decision/Branch Coverage

Decision coverage aims to execute each decision (branch) in the code in both true and false outcomes.

Example: Decision Coverage
function calculateShipping(order) {
  let shippingCost = 0;
  
  // Decision 1: Express shipping?
  if (order.expressShipping) {
    shippingCost = 15;
  } else {
    shippingCost = 5;
  }
  
  // Decision 2: International shipping?
  if (order.international) {
    shippingCost += 10;
  }
  
  // Decision 3: Free shipping for orders over $100?
  if (order.total > 100 && !order.expressShipping) {
    shippingCost = 0;
  }
  
  return shippingCost;
}

// Tests for decision coverage
describe('Calculate Shipping - Decision Coverage', () => {
  test('Regular domestic shipping', () => {
    const order = {
      expressShipping: false,
      international: false,
      total: 50
    };
    
    expect(calculateShipping(order)).toBe(5);
    // Decision 1: false, Decision 2: false, Decision 3: false
  });
  
  test('Express domestic shipping', () => {
    const order = {
      expressShipping: true,
      international: false,
      total: 50
    };
    
    expect(calculateShipping(order)).toBe(15);
    // Decision 1: true, Decision 2: false, Decision 3: false (condition is false)
  });
  
  test('Regular international shipping', () => {
    const order = {
      expressShipping: false,
      international: true,
      total: 50
    };
    
    expect(calculateShipping(order)).toBe(15); // 5 + 10
    // Decision 1: false, Decision 2: true, Decision 3: false
  });
  
  test('Free shipping for large order', () => {
    const order = {
      expressShipping: false,
      international: false,
      total: 150
    };
    
    expect(calculateShipping(order)).toBe(0);
    // Decision 1: false, Decision 2: false, Decision 3: true
  });
  
  // These tests achieve 100% decision coverage
});

Condition Coverage

Condition coverage aims to execute each condition in a decision in both true and false outcomes.

Example: Condition vs. Decision Coverage

For the condition: if (A && B)

  • Decision Coverage: Need tests where (A && B) is true and false
  • Condition Coverage: Need tests where A is true and false, and B is true and false

Test cases for full condition coverage:

  1. A = true, B = true (decision = true)
  2. A = true, B = false (decision = false)
  3. A = false, B = true (decision = false)
  4. A = false, B = false (decision = false)

Experience-Based Techniques

Experience-based techniques rely on the tester's knowledge, skills, and experience to design tests.

Error Guessing

Error guessing involves anticipating where defects might be based on experience and intuition.

Example: Error Guessing for a Login Form

Based on experience, a tester might try:

  • Entering a valid username but incorrect password
  • Entering a username with special characters
  • Leaving fields blank and submitting
  • Entering extremely long values
  • Using SQL injection patterns in input fields
  • Trying to bypass client-side validation

Exploratory Testing

Exploratory testing is simultaneous learning, test design, and test execution.

Example: Exploratory Testing Session

Charter: Explore the shopping cart functionality to discover potential issues with adding, removing, and updating items.

Time Box: 60 minutes

Approach:

  1. Start with basic operations (add, remove, update quantity)
  2. Try edge cases (add maximum items, set quantity to zero)
  3. Explore interactions with other features (wishlist, save for later)
  4. Test across different devices and browsers

Findings: Document issues, observations, and questions during the session

Coverage Criteria

Coverage criteria help measure the thoroughness of testing and identify areas that need more tests.

Types of Coverage

  • Requirements Coverage - Percentage of requirements covered by tests
  • Code Coverage - Percentage of code covered by tests (statement, decision, etc.)
  • Risk Coverage - Percentage of identified risks covered by tests
  • Feature Coverage - Percentage of features covered by tests

  • Java: JaCoCo, Cobertura
  • Python: Coverage.py, pytest-cov
  • C#: NCover, dotCover

Designing Effective Test Cases

Effective test cases are clear, concise, and designed to find defects efficiently.

Test Case Structure

A well-structured test case typically includes:

  • Identifier - Unique ID for the test case
  • Title - Brief description of what is being tested
  • Preconditions - Required state before test execution
  • Test Steps - Specific actions to perform
  • Expected Results - What should happen if the system works correctly
  • Postconditions - Expected state after test execution
  • Traceability - Links to requirements or other artifacts

Example: Well-Structured Test Case

ID: TC-LOGIN-001

Title: Verify successful login with valid credentials

Preconditions:

  • User has a valid account
  • User is not already logged in
  • Login page is accessible

Test Steps:

  1. Navigate to the login page
  2. Enter valid username
  3. Enter valid password
  4. Click the "Login" button

Expected Results:

  • User is successfully logged in
  • User is redirected to the dashboard
  • Welcome message displays the user's name

Postconditions:

  • User session is created
  • Login activity is recorded in the system logs

Traceability: REQ-AUTH-001, REQ-AUTH-005

Test Case Design Best Practices

  • Be Specific - Provide clear, detailed steps and expected results
  • One Purpose - Each test case should verify one specific thing
  • Independence - Test cases should not depend on each other
  • Repeatability - Test cases should produce the same results when repeated
  • Maintainability - Design test cases that are easy to update
  • Traceability - Link test cases to requirements or user stories
  • Prioritization - Focus on critical functionality and high-risk areas
Example: Selenium Test Case in JavaScript
const { Builder, By, until } = require('selenium-webdriver');
const assert = require('assert');

describe('Login Functionality', function() {
  let driver;
  
  before(async function() {
    // Set up the WebDriver
    driver = await new Builder().forBrowser('chrome').build();
  });
  
  after(async function() {
    // Quit the WebDriver
    await driver.quit();
  });
  
  it('should login successfully with valid credentials', async function() {
    // Precondition: Navigate to the login page
    await driver.get('https://example.com/login');
    
    // Step 1: Enter valid username
    await driver.findElement(By.id('username')).sendKeys('testuser');
    
    // Step 2: Enter valid password
    await driver.findElement(By.id('password')).sendKeys('Password123');
    
    // Step 3: Click the login button
    await driver.findElement(By.id('login-button')).click();
    
    // Expected Result 1: User is redirected to dashboard
    await driver.wait(until.urlContains('/dashboard'), 5000);
    
    // Expected Result 2: Welcome message displays user's name
    const welcomeMessage = await driver.findElement(By.id('welcome-message')).getText();
    assert.ok(welcomeMessage.includes('testuser'), 'Welcome message should include username');
    
    // Expected Result 3: Logout button is visible (indicating logged-in state)
    const logoutButton = await driver.findElement(By.id('logout-button'));
    assert.ok(await logoutButton.isDisplayed(), 'Logout button should be visible');
  });
  
  it('should show error message with invalid credentials', async function() {
    // Precondition: Navigate to the login page
    await driver.get('https://example.com/login');
    
    // Step 1: Enter valid username
    await driver.findElement(By.id('username')).sendKeys('testuser');
    
    // Step 2: Enter invalid password
    await driver.findElement(By.id('password')).sendKeys('WrongPassword');
    
    // Step 3: Click the login button
    await driver.findElement(By.id('login-button')).click();
    
    // Expected Result 1: Error message is displayed
    const errorMessage = await driver.findElement(By.id('error-message')).getText();
    assert.equal(errorMessage, 'Invalid username or password', 'Error message should indicate invalid credentials');
    
    // Expected Result 2: User remains on login page
    const currentUrl = await driver.getCurrentUrl();
    assert.ok(currentUrl.includes('/login'), 'User should remain on login page');
  });
});

Real-World Example: Spotify's Test Design

Spotify has developed a sophisticated approach to test design that balances automation and manual testing.

Key Aspects of Spotify's Approach

  • Testing Quadrants - Categorizing tests based on business/technology focus and supporting team/critiquing product
  • Automated Testing - Extensive use of unit, integration, and end-to-end tests
  • Exploratory Testing - Structured exploratory testing sessions
  • Test Data Management - Sophisticated approach to generating and managing test data
  • Continuous Testing - Testing integrated throughout the development process

Lessons from Spotify

  • Balance different types of testing based on context
  • Invest in test automation infrastructure
  • Empower teams to own their testing strategy
  • Use data to drive testing decisions
  • Continuously improve testing practices
