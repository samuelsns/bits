Week 3: Why Do We Test Software?

Learning Objectives

By the end of this week, you will be able to:

  • Understand the fundamental goals and objectives of software testing
  • Identify when software goes bad and recognize common failure patterns
  • Explain the economic benefits of testing and quality assurance
  • Describe the relationship between testing and software quality
  • Recognize the different perspectives on software testing
  • Apply basic testing principles to evaluate software quality

Introduction to Software Testing Goals

Software testing is a critical activity in the software development lifecycle that helps ensure the quality, reliability, and performance of software products. But what exactly are we trying to achieve through testing?

Primary Goals of Software Testing

  • Verification and Validation - Ensuring the software meets requirements (verification) and fulfills its intended purpose (validation)
  • Defect Detection - Finding bugs, errors, and issues before they reach users
  • Quality Assessment - Evaluating the overall quality and readiness of the software
  • Risk Reduction - Minimizing the risk of failures in production
  • Compliance - Ensuring the software meets regulatory and industry standards
  • User Satisfaction - Delivering a product that meets user expectations

Software Quality Attributes

Testing helps ensure that software exhibits these key quality attributes:

  • Functionality - Does the software do what it's supposed to do?
  • Reliability - Does it work consistently without failures?
  • Usability - Is it easy to use and understand?
  • Efficiency - Does it use resources efficiently?
  • Maintainability - Is it easy to modify and enhance?
  • Portability - Can it work in different environments?
  • Security - Is it protected against vulnerabilities and attacks?
  • Scalability - Can it handle growth in users or data?

When Software Goes Bad

Understanding software failures helps us appreciate the importance of testing and identify what to test for.

Notable Software Failures

Ariane 5 Rocket Explosion (1996)

What Happened: The Ariane 5 rocket exploded just 40 seconds after launch due to a software error.

Root Cause: A 64-bit floating-point number was converted to a 16-bit integer, causing an overflow error.

Cost: Approximately $370 million

Lesson: Proper testing of reused code in new contexts is essential. The code was reused from Ariane 4 but not tested under Ariane 5's different flight parameters.

Therac-25 Radiation Overdoses (1985-1987)

What Happened: A radiation therapy machine delivered massive overdoses to patients, resulting in deaths and serious injuries.

Root Cause: Race conditions in the software and removal of hardware safety interlocks.

Cost: Multiple lives and millions in settlements

Lesson: Safety-critical systems require rigorous testing, including edge cases and concurrent operations.

Y2K Bug (2000)

What Happened: Concern that date-related processing would fail when the year changed from 1999 to 2000.

Root Cause: Years represented as two digits (99 instead of 1999) to save memory.

Cost: $300+ billion in remediation efforts worldwide

Lesson: Consider long-term implications of design decisions and test for future scenarios.

Knight Capital Trading Loss (2012)

What Happened: A trading firm lost $440 million in 45 minutes due to a software deployment error.

Root Cause: Old code was reactivated during a deployment, and one server wasn't updated with new code.

Cost: $440 million in 45 minutes

Lesson: Deployment processes need thorough testing, and systems should have safeguards against runaway processes.

Common Types of Software Failures

  • Functional Failures - Software doesn't perform its intended functions correctly
  • Performance Issues - Slow response times, high resource usage
  • Security Vulnerabilities - Weaknesses that can be exploited
  • Usability Problems - Difficult or confusing user interfaces
  • Compatibility Issues - Problems with specific environments or configurations
  • Data Handling Errors - Incorrect processing, storage, or retrieval of data
  • Integration Failures - Components don't work together properly
  • Deployment Problems - Issues during installation or updates
Example: A Simple Bug That Could Cause Major Problems
// Function to transfer money between accounts
function transferMoney(fromAccount, toAccount, amount) {
  // Check if the source account has sufficient funds
  if (fromAccount.balance >= amount) {
    // Deduct from source account
    fromAccount.balance -= amount;
    
    // What if there's an error or system crash here?
    // The money would be deducted but never added to the destination!
    
    // Add to destination account
    toAccount.balance += amount;
    
    return true;
  }
  
  return false;
}

// A more robust implementation with transaction handling
function safeTransferMoney(fromAccount, toAccount, amount) {
  // Start a transaction
  try {
    db.beginTransaction();
    
    // Check if the source account has sufficient funds
    if (fromAccount.balance < amount) {
      throw new Error("Insufficient funds");
    }
    
    // Deduct from source account
    fromAccount.balance -= amount;
    db.update(fromAccount);
    
    // Add to destination account
    toAccount.balance += amount;
    db.update(toAccount);
    
    // Commit the transaction
    db.commitTransaction();
    return true;
  } catch (error) {
    // Rollback the transaction if any error occurs
    db.rollbackTransaction();
    console.error("Transfer failed:", error.message);
    return false;
  }
}

The Economics of Testing

Testing is not just a technical activity; it has significant economic implications for software development.

The Cost of Quality

The cost of quality includes both the cost of achieving good quality and the cost of poor quality:

  • Prevention Costs - Training, process improvement, tool selection
  • Appraisal Costs - Testing, reviews, inspections
  • Internal Failure Costs - Debugging, fixing, retesting
  • External Failure Costs - Support, warranties, lost business, reputation damage

Return on Investment (ROI) of Testing

Testing provides significant ROI through:

  • Reduced Maintenance Costs - Fewer bugs to fix after release
  • Improved Customer Satisfaction - Higher quality leads to happier users
  • Reduced Support Costs - Fewer support tickets and calls
  • Faster Time to Market - Fewer delays due to critical bugs
  • Reduced Risk - Lower likelihood of catastrophic failures
  • Better Reputation - Higher quality products build trust

Testing Economics in Different Contexts

The economics of testing vary based on the context:

Context Testing Focus Economic Considerations
Safety-Critical Systems Exhaustive testing, formal verification High upfront cost justified by preventing catastrophic failures
Enterprise Software Functional correctness, reliability, security Balance between quality and time-to-market
Consumer Apps Usability, performance, core functionality Focus on user-facing features, rapid release cycles
Startups Critical features, minimum viable product Limited resources, prioritize testing for key functionality

Testing and Software Quality

Testing is a key mechanism for ensuring and improving software quality, but it's important to understand both its capabilities and limitations.

What Testing Can and Cannot Do

Testing Can Testing Cannot
Demonstrate the presence of defects Prove the absence of defects
Reduce the risk of problems Eliminate all risks
Provide information about quality Improve quality by itself
Verify specific behaviors Verify all possible behaviors
Find many types of defects Find all types of defects
Validate against requirements Validate against unstated expectations

Testing as Part of Quality Assurance

Testing is one component of a comprehensive quality assurance strategy:

  • Quality Planning - Defining quality objectives and approaches
  • Quality Control - Monitoring and measuring quality
  • Quality Assurance - Ensuring processes are followed
  • Testing - Verifying and validating the software
  • Quality Improvement - Enhancing processes and practices
Example: Simple Unit Test with Jest (JavaScript)
// Function to be tested
function calculateTotal(items, discount = 0) {
  if (!Array.isArray(items)) {
    throw new Error('Items must be an array');
  }
  
  if (discount < 0 || discount > 1) {
    throw new Error('Discount must be between 0 and 1');
  }
  
  const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const total = subtotal * (1 - discount);
  
  return parseFloat(total.toFixed(2));
}

// Unit tests using Jest
describe('calculateTotal function', () => {
  test('calculates total correctly without discount', () => {
    const items = [
      { name: 'Item 1', price: 10, quantity: 2 },
      { name: 'Item 2', price: 15, quantity: 1 }
    ];
    
    expect(calculateTotal(items)).toBe(35);
  });
  
  test('calculates total correctly with discount', () => {
    const items = [
      { name: 'Item 1', price: 10, quantity: 2 },
      { name: 'Item 2', price: 15, quantity: 1 }
    ];
    
    expect(calculateTotal(items, 0.1)).toBe(31.5); // 35 - 10% = 31.5
  });
  
  test('handles empty array', () => {
    expect(calculateTotal([])).toBe(0);
  });
  
  test('throws error for invalid items', () => {
    expect(() => calculateTotal('not an array')).toThrow('Items must be an array');
  });
  
  test('throws error for invalid discount', () => {
    const items = [{ name: 'Item 1', price: 10, quantity: 1 }];
    
    expect(() => calculateTotal(items, -0.1)).toThrow('Discount must be between 0 and 1');
    expect(() => calculateTotal(items, 1.1)).toThrow('Discount must be between 0 and 1');
  });
  
  test('rounds to 2 decimal places', () => {
    const items = [{ name: 'Item 1', price: 10.999, quantity: 1 }];
    
    expect(calculateTotal(items)).toBe(11);
  });
});

Quality Models and Standards

Various models and standards define software quality:

  • ISO/IEC 25010 - Product quality model defining quality characteristics
  • ISO/IEC 29119 - Software testing standards
  • CMMI (Capability Maturity Model Integration) - Process improvement approach
  • TMMi (Test Maturity Model integration) - Framework for test process improvement
  • TPI (Test Process Improvement) - Model for improving test processes

Perspectives on Software Testing

Different stakeholders have different perspectives on software testing, which influence testing priorities and approaches.

Stakeholder Perspectives

  • Developers - Focus on code correctness, maintainability, technical quality
  • Testers - Focus on finding defects, validating requirements, assessing quality
  • Project Managers - Focus on schedule, budget, risk management
  • Business Analysts - Focus on requirements coverage, business value
  • End Users - Focus on usability, functionality, performance
  • Operations - Focus on reliability, security, supportability
  • Executives - Focus on ROI, market impact, competitive advantage

Testing Mindsets

Different mindsets influence how testing is approached:

  • Verification Mindset - "Does the software meet the requirements?"
  • Validation Mindset - "Does the software solve the user's problem?"
  • Destructive Mindset - "How can I break this software?"
  • Constructive Mindset - "How can I help improve this software?"
  • Analytical Mindset - "What's happening under the hood?"
  • User-Centered Mindset - "What would a user experience?"

Principles of Software Testing

The International Software Testing Qualifications Board (ISTQB) defines seven fundamental principles of software testing:

1. Testing shows the presence of defects, not their absence

Testing can show that defects are present, but cannot prove that there are no defects. Even after thorough testing, we cannot guarantee that software is defect-free.

2. Exhaustive testing is impossible

Testing everything (all combinations of inputs and preconditions) is not feasible except for trivial cases. Instead of exhaustive testing, we use risk analysis, test techniques, and priorities to focus testing efforts.

3. Early testing saves time and money

Testing activities should start as early as possible in the software development lifecycle. Finding defects early is less expensive to fix than finding them later.

4. Defects cluster together

A small number of modules usually contains most of the defects. This "Pareto principle" (80/20 rule) helps focus testing efforts on the most defect-prone areas.

5. Beware of the pesticide paradox

If the same tests are repeated over and over, they will eventually no longer find new defects. To overcome this, tests need to be regularly reviewed and revised.

6. Testing is context dependent

Testing is done differently in different contexts. For example, safety-critical software is tested differently from an e-commerce site.

7. Absence of errors is a fallacy

Finding and fixing defects does not help if the system is unusable or does not fulfill user needs and expectations.

Applying Testing Principles

These principles guide effective testing practices:

  • Risk-Based Testing - Focus testing efforts on the most critical areas
  • Test Design Techniques - Use systematic approaches to design effective tests
  • Continuous Testing - Integrate testing throughout the development process
  • Test Independence - Have tests performed by someone other than the developer
  • Test Automation - Automate repetitive tests to improve efficiency
  • Defect Management - Track and manage defects systematically

Real-World Example: The Boeing 737 MAX Software Failure

The Boeing 737 MAX case provides a compelling example of the importance of thorough software testing and the potential consequences of inadequate testing.

Background

In 2018 and 2019, two Boeing 737 MAX aircraft crashed, killing 346 people. The crashes were linked to a software system called MCAS (Maneuvering Characteristics Augmentation System).

What Went Wrong

  • Single Point of Failure - MCAS relied on a single angle-of-attack sensor without redundancy
  • Inadequate Testing - The system wasn't tested under all possible failure conditions
  • Poor Documentation - Pilots weren't adequately informed about the system
  • Regulatory Issues - FAA delegated too much of the safety assessment to Boeing
  • Development Pressure - Schedule pressure may have compromised thorough testing

Testing Lessons

  • Test for Failure Conditions - Test how systems behave when components fail
  • Consider Human Factors - Test how users will interact with the system
  • Independent Verification - Have critical systems verified by independent testers
  • Thorough Risk Analysis - Identify and mitigate potential risks
  • Resist Schedule Pressure - Don't compromise safety for deadlines
