Week 8: Infrastructure as Code

Learning Objectives

By the end of this week, you will be able to:

  • Understand the principles and benefits of Infrastructure as Code (IaC)
  • Compare different IaC tools and approaches
  • Implement infrastructure provisioning using Terraform
  • Apply best practices for managing infrastructure code
  • Test infrastructure code effectively
  • Integrate IaC into CI/CD pipelines

Introduction to Infrastructure as Code

Infrastructure as Code (IaC) is the practice of managing and provisioning computing infrastructure through machine-readable definition files, rather than physical hardware configuration or interactive configuration tools.

Key Principles of Infrastructure as Code

  • Reproducibility - Infrastructure can be recreated consistently from the same code
  • Idempotence - Running the same code multiple times produces the same result
  • Declarative - Define what the infrastructure should be, not how to create it
  • Version Control - Infrastructure code is stored in version control systems
  • Testability - Infrastructure code can be tested like application code
  • Automation - Infrastructure changes are applied automatically

Benefits of Infrastructure as Code

  • Speed and Consistency - Rapidly deploy infrastructure with consistent results
  • Documentation - Code serves as documentation for the infrastructure
  • Version Control - Track changes to infrastructure over time
  • Collaboration - Multiple team members can work on infrastructure
  • Disaster Recovery - Quickly rebuild infrastructure in case of failure
  • Cost Reduction - Optimize resource usage and reduce manual effort
  • Risk Reduction - Minimize human error in infrastructure changes

IaC Tools and Approaches

There are various tools and approaches for implementing Infrastructure as Code, each with its own strengths and use cases.

Types of IaC Tools

  • Infrastructure Provisioning Tools - Create and manage infrastructure resources (e.g., Terraform, CloudFormation)
  • Configuration Management Tools - Install and manage software on existing infrastructure (e.g., Ansible, Chef, Puppet)
  • Server Templating Tools - Create machine images with pre-installed software (e.g., Packer, Docker)
  • Container Orchestration Tools - Manage containerized applications (e.g., Kubernetes, Docker Swarm)

Popular IaC Tools

Tool Type Language Approach Best For
Terraform Provisioning HCL Declarative Multi-cloud infrastructure provisioning
AWS CloudFormation Provisioning JSON/YAML Declarative AWS-specific infrastructure
Azure Resource Manager Provisioning JSON/Bicep Declarative Azure-specific infrastructure
Google Cloud Deployment Manager Provisioning YAML/Python/Jinja2 Declarative GCP-specific infrastructure
Ansible Configuration Management YAML Procedural Configuration management, application deployment
Chef Configuration Management Ruby Procedural Complex configuration management
Puppet Configuration Management Puppet DSL Declarative Large-scale configuration management
Packer Server Templating JSON/HCL Declarative Creating machine images

Declarative vs. Procedural Approaches

  • Declarative - Define the desired end state, not how to get there
  • Procedural - Define the steps to achieve the desired state

Most modern IaC tools favor a declarative approach, which provides better idempotence and reduces the risk of configuration drift.

Infrastructure Provisioning with Terraform

Terraform is a popular open-source infrastructure provisioning tool that supports multiple cloud providers and services.

Key Terraform Concepts

  • Providers - Plugins that interact with cloud providers or services
  • Resources - Infrastructure components managed by Terraform
  • Data Sources - Read-only information about existing infrastructure
  • Variables - Input parameters for Terraform configurations
  • Outputs - Return values from Terraform configurations
  • Modules - Reusable Terraform configurations
  • State - Terraform's record of managed infrastructure

Terraform Workflow

  1. Write - Create Terraform configuration files
  2. Initialize - Run terraform init to download providers
  3. Plan - Run terraform plan to preview changes
  4. Apply - Run terraform apply to create or update infrastructure
  5. Destroy - Run terraform destroy to remove infrastructure
Example: Terraform Configuration for AWS Infrastructure
# main.tf

# Configure the AWS Provider
provider "aws" {
  region = var.aws_region
}

# Create a VPC
resource "aws_vpc" "main" {
  cidr_block = var.vpc_cidr
  
  tags = {
    Name = "${var.project_name}-vpc"
    Environment = var.environment
  }
}

# Create public subnets
resource "aws_subnet" "public" {
  count = length(var.public_subnet_cidrs)
  
  vpc_id            = aws_vpc.main.id
  cidr_block        = var.public_subnet_cidrs[count.index]
  availability_zone = var.availability_zones[count.index]
  
  map_public_ip_on_launch = true
  
  tags = {
    Name = "${var.project_name}-public-subnet-${count.index + 1}"
    Environment = var.environment
  }
}

# Create private subnets
resource "aws_subnet" "private" {
  count = length(var.private_subnet_cidrs)
  
  vpc_id            = aws_vpc.main.id
  cidr_block        = var.private_subnet_cidrs[count.index]
  availability_zone = var.availability_zones[count.index]
  
  tags = {
    Name = "${var.project_name}-private-subnet-${count.index + 1}"
    Environment = var.environment
  }
}

# Create Internet Gateway
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id
  
  tags = {
    Name = "${var.project_name}-igw"
    Environment = var.environment
  }
}

# Create route table for public subnets
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }
  
  tags = {
    Name = "${var.project_name}-public-route-table"
    Environment = var.environment
  }
}

# Associate public subnets with public route table
resource "aws_route_table_association" "public" {
  count = length(var.public_subnet_cidrs)
  
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

# Create security group for web servers
resource "aws_security_group" "web" {
  name        = "${var.project_name}-web-sg"
  description = "Security group for web servers"
  vpc_id      = aws_vpc.main.id
  
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = [var.admin_cidr]
  }
  
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  tags = {
    Name = "${var.project_name}-web-sg"
    Environment = var.environment
  }
}

# Create EC2 instances
resource "aws_instance" "web" {
  count = var.instance_count
  
  ami           = var.ami_id
  instance_type = var.instance_type
  subnet_id     = aws_subnet.public[count.index % length(var.public_subnet_cidrs)].id
  
  vpc_security_group_ids = [aws_security_group.web.id]
  key_name               = var.key_name
  
  user_data = <<-EOF
              #!/bin/bash
              echo "Hello, World" > index.html
              nohup python -m SimpleHTTPServer 80 &
              EOF
  
  tags = {
    Name = "${var.project_name}-web-${count.index + 1}"
    Environment = var.environment
  }
}

# Create an Application Load Balancer
resource "aws_lb" "web" {
  name               = "${var.project_name}-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.web.id]
  subnets            = aws_subnet.public[*].id
  
  tags = {
    Name = "${var.project_name}-alb"
    Environment = var.environment
  }
}

# Create ALB target group
resource "aws_lb_target_group" "web" {
  name     = "${var.project_name}-tg"
  port     = 80
  protocol = "HTTP"
  vpc_id   = aws_vpc.main.id
  
  health_check {
    path                = "/"
    port                = "traffic-port"
    healthy_threshold   = 2
    unhealthy_threshold = 2
    timeout             = 3
    interval            = 30
  }
}

# Attach instances to target group
resource "aws_lb_target_group_attachment" "web" {
  count = var.instance_count
  
  target_group_arn = aws_lb_target_group.web.arn
  target_id        = aws_instance.web[count.index].id
  port             = 80
}

# Create ALB listener
resource "aws_lb_listener" "web" {
  load_balancer_arn = aws_lb.web.arn
  port              = 80
  protocol          = "HTTP"
  
  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.web.arn
  }
}

# Output the ALB DNS name
output "alb_dns_name" {
  value = aws_lb.web.dns_name
}
Example: Terraform Variables
# variables.tf

variable "aws_region" {
  description = "AWS region to deploy resources"
  type        = string
  default     = "us-west-2"
}

variable "project_name" {
  description = "Name of the project"
  type        = string
  default     = "demo"
}

variable "environment" {
  description = "Environment (e.g., dev, staging, prod)"
  type        = string
  default     = "dev"
}

variable "vpc_cidr" {
  description = "CIDR block for the VPC"
  type        = string
  default     = "10.0.0.0/16"
}

variable "public_subnet_cidrs" {
  description = "CIDR blocks for the public subnets"
  type        = list(string)
  default     = ["10.0.1.0/24", "10.0.2.0/24"]
}

variable "private_subnet_cidrs" {
  description = "CIDR blocks for the private subnets"
  type        = list(string)
  default     = ["10.0.3.0/24", "10.0.4.0/24"]
}

variable "availability_zones" {
  description = "Availability zones for the subnets"
  type        = list(string)
  default     = ["us-west-2a", "us-west-2b"]
}

variable "instance_count" {
  description = "Number of EC2 instances to create"
  type        = number
  default     = 2
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"
}

variable "ami_id" {
  description = "AMI ID for the EC2 instances"
  type        = string
  default     = "ami-0c55b159cbfafe1f0"  # Amazon Linux 2 in us-west-2
}

variable "key_name" {
  description = "Name of the SSH key pair"
  type        = string
  default     = "my-key-pair"
}

variable "admin_cidr" {
  description = "CIDR block for SSH access"
  type        = string
  default     = "0.0.0.0/0"  # Not recommended for production
}

IaC Best Practices

Following best practices for Infrastructure as Code helps ensure that your infrastructure is reliable, maintainable, and secure.

Version Control

  • Store all infrastructure code in version control
  • Use branches and pull requests for changes
  • Require code reviews before merging
  • Tag releases for important infrastructure versions

Modularity and Reusability

  • Create reusable modules for common infrastructure patterns
  • Use consistent naming conventions
  • Parameterize modules with variables
  • Document module inputs, outputs, and usage

State Management

  • Store state files in a secure, shared location
  • Use state locking to prevent concurrent modifications
  • Separate state files for different environments
  • Backup state files regularly

Security

  • Never store secrets in infrastructure code
  • Use secret management tools (e.g., HashiCorp Vault, AWS Secrets Manager)
  • Apply the principle of least privilege
  • Regularly audit and rotate credentials

Testing

  • Validate syntax and formatting
  • Run static analysis to catch common issues
  • Write unit tests for modules
  • Perform integration tests in isolated environments

Documentation

  • Document the purpose and usage of infrastructure code
  • Include README files in repositories
  • Generate documentation from code when possible
  • Keep documentation up-to-date with code changes

Testing Infrastructure Code

Testing infrastructure code is essential for ensuring that your infrastructure is reliable and behaves as expected.

Types of Infrastructure Tests

  • Static Analysis - Check syntax, formatting, and best practices
  • Unit Tests - Test individual modules or components
  • Integration Tests - Test interactions between components
  • End-to-End Tests - Test the complete infrastructure
  • Compliance Tests - Verify that infrastructure meets security and compliance requirements

Testing Tools for Terraform

  • terraform validate - Check syntax and consistency
  • terraform fmt - Format code according to conventions
  • tflint - Linter for Terraform code
  • Terratest - Go library for testing Terraform code
  • kitchen-terraform - Test Kitchen plugin for Terraform
  • Terraform Cloud/Enterprise - Policy as Code with Sentinel
Example: Terratest for Testing Terraform Code
package test

import (
	"testing"
	"time"

	"github.com/gruntwork-io/terratest/modules/terraform"
	"github.com/stretchr/testify/assert"
)

func TestTerraformWebServer(t *testing.T) {
	// Terraform options with the path to the Terraform code
	terraformOptions := &terraform.Options{
		// The path to where the Terraform code is located
		TerraformDir: "../examples/web-server",

		// Variables to pass to the Terraform code using -var options
		Vars: map[string]interface{}{
			"aws_region":    "us-west-2",
			"instance_type": "t2.micro",
			"project_name":  "terratest",
			"environment":   "test",
		},

		// Environment variables to set when running Terraform
		EnvVars: map[string]string{
			"AWS_DEFAULT_REGION": "us-west-2",
		},
	}

	// Clean up resources at the end of the test
	defer terraform.Destroy(t, terraformOptions)

	// Initialize and apply the Terraform code
	terraform.InitAndApply(t, terraformOptions)

	// Get the output variables
	albDnsName := terraform.Output(t, terraformOptions, "alb_dns_name")

	// Verify the ALB DNS name is not empty
	assert.NotEmpty(t, albDnsName, "ALB DNS name should not be empty")

	// Wait for the ALB to become available
	time.Sleep(30 * time.Second)

	// Add HTTP request to verify the ALB is working
	// ...
}

Integrating IaC into CI/CD Pipelines

Integrating Infrastructure as Code into CI/CD pipelines enables automated testing and deployment of infrastructure changes.

CI/CD for Infrastructure Code

  1. Commit - Push infrastructure code changes to version control
  2. Build - Run static analysis and validation
  3. Test - Run unit and integration tests
  4. Plan - Generate and review Terraform plan
  5. Approve - Manually approve changes (optional)
  6. Apply - Apply changes to infrastructure
  7. Verify - Verify that infrastructure is working as expected
Example: GitHub Actions Workflow for Terraform
name: Terraform CI/CD

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  terraform:
    name: Terraform
    runs-on: ubuntu-latest
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v3
    
    - name: Setup Terraform
      uses: hashicorp/setup-terraform@v2
      with:
        terraform_version: 1.0.0
    
    - name: Terraform Format
      id: fmt
      run: terraform fmt -check
    
    - name: Terraform Init
      id: init
      run: terraform init
    
    - name: Terraform Validate
      id: validate
      run: terraform validate
    
    - name: Terraform Plan
      id: plan
      if: github.event_name == 'pull_request'
      run: terraform plan -no-color
      continue-on-error: true
    
    - name: Update Pull Request
      uses: actions/github-script@v6
      if: github.event_name == 'pull_request'
      env:
        PLAN: "terraform\n${{ steps.plan.outputs.stdout }}"
      with:
        github-token: ${{ secrets.GITHUB_TOKEN }}
        script: |
          const output = `#### Terraform Format and Style üñå\`${{ steps.fmt.outcome }}\`

          #### Terraform Validation ü§ñ\`${{ steps.validate.outcome }}\`

Benefits of CI/CD for Infrastructure

  • Consistency - Infrastructure changes follow the same process as application changes
  • Visibility - Changes are visible to the team through the CI/CD pipeline
  • Quality - Automated testing improves infrastructure quality
  • Speed - Automation reduces the time to implement changes
  • Auditability - CI/CD provides a record of infrastructure changes

Real-World Example: Capital One's IaC Approach

Capital One has embraced Infrastructure as Code as part of its cloud transformation journey.

Key Aspects of Capital One's IaC Approach

  • Cloud-First Strategy - Moving all applications to the cloud
  • Terraform for Infrastructure - Using Terraform for infrastructure provisioning
  • Modular Approach - Creating reusable Terraform modules
  • Automated Testing - Implementing comprehensive testing for infrastructure code
  • CI/CD Integration - Integrating infrastructure changes into CI/CD pipelines
  • Security and Compliance - Enforcing security and compliance through code

Lessons from Capital One

  • Start with a clear strategy and goals
  • Invest in training and culture change
  • Build a library of reusable modules
  • Automate testing and deployment
  • Prioritize security and compliance
  • Continuously improve and refine the approach
