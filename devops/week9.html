Week 9: Configuration Management

Learning Objectives

By the end of this week, you will be able to:

  • Understand the principles and benefits of configuration management
  • Compare different configuration management tools
  • Implement server configuration using Ansible
  • Apply configuration management best practices
  • Integrate configuration management with infrastructure provisioning
  • Manage secrets and sensitive data securely

Introduction to Configuration Management

Configuration Management (CM) is the practice of handling changes to a system in a way that maintains system integrity over time. In the context of DevOps, it involves automating the setup and maintenance of systems to ensure they're in a desired, consistent state.

Key Principles of Configuration Management

  • Consistency - Systems are configured in a consistent, predictable manner
  • Idempotence - Running the same configuration multiple times produces the same result
  • Automation - Configuration changes are applied automatically
  • Version Control - Configuration code is stored in version control systems
  • Documentation - Configuration serves as documentation for the system
  • Testing - Configuration can be tested before applying to production

Benefits of Configuration Management

  • Reduced Configuration Drift - Systems remain in their desired state
  • Faster Recovery - Systems can be quickly rebuilt after failures
  • Scalability - New systems can be configured automatically
  • Compliance - Systems can be configured to meet security and compliance requirements
  • Collaboration - Teams can work together on system configuration
  • Auditability - Changes to configuration are tracked and documented

Discussion Question

How does configuration management differ from infrastructure provisioning? In what ways do they complement each other in a DevOps workflow?

Configuration Management Tools

There are various tools for implementing configuration management, each with its own approach and strengths.

Popular Configuration Management Tools

Tool Approach Language Architecture Best For
Ansible Procedural YAML Agentless (SSH) Simple, ad-hoc configuration management
Chef Procedural Ruby Agent-based Complex, large-scale environments
Puppet Declarative Puppet DSL Agent-based Enterprise environments with strict compliance
SaltStack Declarative/Procedural YAML/Python Agent-based High-speed execution at scale

Choosing a Configuration Management Tool

When selecting a configuration management tool, consider the following factors:

  • Learning Curve - How easy is it to learn and use?
  • Community and Support - Is there a strong community and good documentation?
  • Integration - Does it integrate well with your existing tools and workflows?
  • Scalability - Can it handle your environment size and complexity?
  • Agent vs. Agentless - Do you prefer an agent-based or agentless approach?
  • Idempotence - Does it ensure idempotent operations?
  • Reporting - Does it provide good visibility into configuration state?

Server Configuration with Ansible

Ansible is a popular configuration management tool known for its simplicity and agentless architecture.

Key Ansible Concepts

  • Inventory - List of hosts to be managed
  • Playbooks - YAML files describing the desired state
  • Tasks - Individual units of configuration
  • Modules - Plugins that perform specific actions
  • Roles - Reusable collections of tasks
  • Variables - Values that can be used in playbooks
  • Templates - Jinja2 templates for generating files
  • Handlers - Tasks that run when notified by other tasks

Ansible Architecture

Ansible uses an agentless architecture, connecting to managed nodes via SSH (for Linux) or WinRM (for Windows). This simplifies deployment and reduces maintenance overhead.

  • Control Node - The system where Ansible is installed
  • Managed Nodes - The systems being configured
  • Inventory - Defines the managed nodes
  • Modules - Executed on managed nodes
  • Plugins - Extend Ansible's functionality
Example: Ansible Inventory File
# inventory.ini

# Web servers
[webservers]
web1.example.com
web2.example.com
web3.example.com

# Database servers
[dbservers]
db1.example.com
db2.example.com

# Load balancers
[loadbalancers]
lb1.example.com
lb2.example.com

# Group of groups
[production:children]
webservers
dbservers
loadbalancers

# Group variables
[webservers:vars]
http_port=80
https_port=443
web_app_version=1.2.3

[dbservers:vars]
db_port=5432
db_version=12
Example: Ansible Playbook for Web Server Configuration
---
# webserver.yml
- name: Configure web servers
  hosts: webservers
  become: yes
  vars:
    app_name: my-web-app
    app_version: "{{ web_app_version | default('1.0.0') }}"
    app_port: "{{ http_port | default(80) }}"
    
  tasks:
    - name: Update apt cache
      apt:
        update_cache: yes
        cache_valid_time: 3600
      when: ansible_os_family == "Debian"
    
    - name: Install required packages
      package:
        name:
          - nginx
          - python3
          - python3-pip
          - git
        state: present
    
    - name: Configure Nginx
      template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/sites-available/{{ app_name }}
        owner: root
        group: root
        mode: '0644'
      notify: restart nginx
    
    - name: Enable Nginx site
      file:
        src: /etc/nginx/sites-available/{{ app_name }}
        dest: /etc/nginx/sites-enabled/{{ app_name }}
        state: link
      notify: restart nginx
    
    - name: Ensure app directory exists
      file:
        path: /var/www/{{ app_name }}
        state: directory
        owner: www-data
        group: www-data
        mode: '0755'
    
    - name: Clone application repository
      git:
        repo: https://github.com/example/{{ app_name }}.git
        dest: /var/www/{{ app_name }}
        version: "{{ app_version }}"
      notify: restart application
    
    - name: Install application dependencies
      pip:
        requirements: /var/www/{{ app_name }}/requirements.txt
        state: present
    
    - name: Configure application
      template:
        src: templates/app_config.j2
        dest: /var/www/{{ app_name }}/config.py
        owner: www-data
        group: www-data
        mode: '0644'
      notify: restart application
    
    - name: Create application service
      template:
        src: templates/app_service.j2
        dest: /etc/systemd/system/{{ app_name }}.service
        owner: root
        group: root
        mode: '0644'
      notify: restart application
    
    - name: Ensure application service is enabled
      systemd:
        name: "{{ app_name }}"
        enabled: yes
        daemon_reload: yes
    
  handlers:
    - name: restart nginx
      systemd:
        name: nginx
        state: restarted
    
    - name: restart application
      systemd:
        name: "{{ app_name }}"
        state: restarted
Example: Nginx Configuration Template (Jinja2)
# templates/nginx.conf.j2
server {
    listen {{ http_port }};
    server_name {{ inventory_hostname }};

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /static {
        alias /var/www/{{ app_name }}/static;
    }
}

Ansible Roles

Roles are a way to organize playbooks and reuse code. A role is a set of tasks, variables, files, templates, and handlers that are stored in a standardized directory structure.

Example: Ansible Role Directory Structure
roles/
└── webserver/
    ├── defaults/
    │   └── main.yml       # Default variables
    ├── files/
    │   └── app.conf       # Static files
    ├── handlers/
    │   └── main.yml       # Handlers
    ├── meta/
    │   └── main.yml       # Role metadata
    ├── tasks/
    │   └── main.yml       # Tasks
    ├── templates/
    │   ├── nginx.conf.j2  # Templates
    │   └── app_config.j2
    └── vars/
        └── main.yml       # Role variables
Example: Using Roles in a Playbook
---
# site.yml
- name: Configure web servers
  hosts: webservers
  become: yes
  roles:
    - common
    - webserver
    - monitoring

- name: Configure database servers
  hosts: dbservers
  become: yes
  roles:
    - common
    - database
    - monitoring

Configuration Management Best Practices

Following best practices for configuration management helps ensure that your systems are reliable, maintainable, and secure.

Version Control

  • Store all configuration code in version control
  • Use branches and pull requests for changes
  • Require code reviews before merging
  • Tag releases for important configuration versions

Modularity and Reusability

  • Create reusable roles or modules for common patterns
  • Use consistent naming conventions
  • Parameterize configurations with variables
  • Document inputs, outputs, and usage

Testing

  • Test configurations in development environments before applying to production
  • Use tools like Molecule for testing Ansible roles
  • Write idempotent configurations that can be run multiple times
  • Include validation steps in your configurations

Security

  • Never store secrets in configuration code
  • Use secret management tools (e.g., Ansible Vault, HashiCorp Vault)
  • Apply the principle of least privilege
  • Regularly audit and rotate credentials

Documentation

  • Document the purpose and usage of configuration code
  • Include README files in repositories
  • Comment complex or non-obvious configurations
  • Keep documentation up-to-date with code changes

Integrating Configuration Management with Infrastructure Provisioning

Configuration management and infrastructure provisioning are complementary practices that work together to create and maintain your infrastructure.

Approaches to Integration

  • Sequential Approach - Provision infrastructure first, then configure it
  • Parallel Approach - Use separate tools for provisioning and configuration
  • Unified Approach - Use a single tool for both provisioning and configuration
  • Hybrid Approach - Use specialized tools but integrate them in a workflow

Terraform and Ansible Integration

A common approach is to use Terraform for infrastructure provisioning and Ansible for configuration management.

Example: Terraform Provisioning with Ansible Configuration
# main.tf

# Provision EC2 instance
resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = var.instance_type
  key_name      = var.key_name
  
  vpc_security_group_ids = [aws_security_group.web.id]
  
  tags = {
    Name = "${var.project_name}-web"
  }
  
  # Generate inventory file for Ansible
  provisioner "local-exec" {
    command = <<-EOT
      echo "[webservers]" > inventory.ini
      echo "${self.public_ip} ansible_user=ubuntu ansible_ssh_private_key_file=${var.private_key_path}" >> inventory.ini
    EOT
  }
  
  # Run Ansible playbook
  provisioner "local-exec" {
    command = "ansible-playbook -i inventory.ini webserver.yml"
  }
}

# Output the instance IP
output "web_ip" {
  value = aws_instance.web.public_ip
}

Benefits of Integration

  • End-to-End Automation - Automate the entire process from provisioning to configuration
  • Consistency - Ensure consistent environments from development to production
  • Efficiency - Reduce manual steps and human error
  • Scalability - Easily scale infrastructure up or down
  • Flexibility - Use the best tool for each job

Managing Secrets and Sensitive Data

Securely managing secrets and sensitive data is a critical aspect of configuration management.

Challenges of Secret Management

  • Secrets should not be stored in version control
  • Secrets need to be accessible to authorized systems and users
  • Secrets should be rotated regularly
  • Access to secrets should be audited

Secret Management Solutions

  • Ansible Vault - Encrypt sensitive data in Ansible playbooks
  • HashiCorp Vault - Dedicated secret management tool
  • AWS Secrets Manager - AWS service for managing secrets
  • Azure Key Vault - Azure service for managing secrets
  • Google Secret Manager - Google Cloud service for managing secrets
Example: Using Ansible Vault
# Create an encrypted file
ansible-vault create secrets.yml

# Edit an encrypted file
ansible-vault edit secrets.yml

# Encrypt an existing file
ansible-vault encrypt secrets.yml

# Decrypt an encrypted file
ansible-vault decrypt secrets.yml

# View an encrypted file
ansible-vault view secrets.yml

# Run a playbook with vault password
ansible-playbook --ask-vault-pass playbook.yml

# Run a playbook with vault password file
ansible-playbook --vault-password-file=vault_pass.txt playbook.yml
Example: Encrypted Variables in Ansible
# secrets.yml (encrypted with Ansible Vault)
---
db_password: supersecretpassword
api_key: abcdef123456
ssl_private_key: |
  -----BEGIN PRIVATE KEY-----
  MIIEvgIBADANBgkqhkiG9w0BAQEFAASCBKgwggSkAgEAAoIBAQC7VJTUt9Us8cKj
  ...
  -----END PRIVATE KEY-----

# Using encrypted variables in a playbook
---
- name: Configure database
  hosts: dbservers
  become: yes
  vars_files:
    - secrets.yml
  
  tasks:
    - name: Set database password
      mysql_user:
        name: myapp
        password: "{{ db_password }}"
        host: localhost
        priv: 'myapp.*:ALL'
        state: present

Best Practices for Secret Management

  • Use a dedicated secret management solution
  • Encrypt secrets at rest and in transit
  • Implement least privilege access to secrets
  • Rotate secrets regularly
  • Audit access to secrets
  • Use different secrets for different environments
  • Never log or display secrets

Real-World Example: Netflix's Configuration Management

Netflix has developed a sophisticated approach to configuration management as part of its cloud-native architecture.

Key Aspects of Netflix's Configuration Management

  • Immutable Infrastructure - Using pre-configured Amazon Machine Images (AMIs)
  • Spinnaker - Continuous delivery platform for managing deployments
  • Archaius - Dynamic property management system
  • Chaos Engineering - Testing configuration resilience with chaos experiments
  • Automated Rollbacks - Automatically rolling back problematic configurations

Lessons from Netflix

  • Embrace immutable infrastructure for consistency
  • Implement dynamic configuration for flexibility
  • Test configuration resilience with chaos engineering
  • Automate rollbacks for quick recovery
  • Invest in tooling for scale
