Week 5: Creating the Foundation of Deployment Pipeline

Learning Objectives

By the end of this week, you will be able to:

  • Understand the concept and components of a deployment pipeline
  • Design a basic deployment pipeline for a software project
  • Implement version control best practices
  • Set up automated build processes
  • Configure basic automated testing in a pipeline
  • Explain the role of infrastructure as code in deployment pipelines

Introduction to Deployment Pipelines

A deployment pipeline is an automated implementation of your application's build, deploy, test, and release process. It provides visibility into the production readiness of your application at each stage of the software delivery process.

Key Benefits of Deployment Pipelines

  • Visibility - Everyone can see the progress of changes through the pipeline
  • Feedback - Fast feedback on the quality and deployability of changes
  • Automation - Consistent, repeatable processes that reduce manual errors
  • Collaboration - Shared responsibility for delivery across teams
  • Confidence - Increased confidence in the release process

Stages of a Typical Deployment Pipeline

  1. Commit Stage - Compile code, run unit tests, static analysis
  2. Automated Acceptance Testing - Verify functionality meets requirements
  3. Additional Testing - Performance, security, compatibility testing
  4. Production Deployment - Deploy to production environments

Version Control Best Practices

Version control is the foundation of any deployment pipeline. It provides a single source of truth for your codebase and enables collaboration among team members.

Git Workflow Strategies

  • Feature Branch Workflow - Create a branch for each feature or bug fix
  • Gitflow - Structured branching model with dedicated branches for features, releases, and hotfixes
  • Trunk-Based Development - Everyone commits to a single branch with frequent integration
  • GitHub Flow - Simplified workflow with feature branches and pull requests

Version Control Best Practices

  • Commit frequently with meaningful commit messages
  • Keep branches short-lived to minimize merge conflicts
  • Use pull requests or code reviews before merging
  • Automate tests to run on every commit
  • Tag releases with version numbers
  • Store configuration in version control (but not secrets)
Example: Git Commands for Feature Branch Workflow
# Create a new feature branch
git checkout -b feature/new-login-page

# Make changes and commit
git add .
git commit -m "Implement new login page design"

# Push branch to remote repository
git push -u origin feature/new-login-page

# Update branch with latest changes from main
git fetch origin
git rebase origin/main

# After pull request is approved, merge to main
git checkout main
git merge --no-ff feature/new-login-page
git push origin main

# Delete the feature branch
git branch -d feature/new-login-page
git push origin --delete feature/new-login-page

Automated Build Processes

Automated builds are a critical component of deployment pipelines, ensuring that code can be compiled and packaged consistently.

Key Components of Build Automation

  • Build Triggers - Events that initiate the build process (e.g., code commits, scheduled builds)
  • Build Tools - Software that compiles and packages code (e.g., Maven, Gradle, npm)
  • Build Servers - Systems that execute and manage builds (e.g., Jenkins, GitHub Actions, CircleCI)
  • Artifacts - Output of the build process (e.g., executables, container images)
  • Artifact Repositories - Storage for build artifacts (e.g., Nexus, Artifactory, Docker Hub)

Setting Up a Basic CI Pipeline with GitHub Actions

GitHub Actions allows you to automate your build, test, and deployment workflows directly in your GitHub repository.

Example: GitHub Actions Workflow for a Node.js Application
name: Node.js CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [14.x, 16.x, 18.x]

    steps:
    - uses: actions/checkout@v3
    - name: Use Node.js ${{ matrix.node-version }}
      uses: actions/setup-node@v3
      with:
        node-version: ${{ matrix.node-version }}
        cache: 'npm'
    - run: npm ci
    - run: npm run build --if-present
    - run: npm test
    
    - name: Upload build artifacts
      uses: actions/upload-artifact@v3
      with:
        name: build-output
        path: dist/

Automated Testing in Pipelines

Automated testing is essential for ensuring the quality of your software as it moves through the deployment pipeline.

Types of Tests in a Pipeline

  • Unit Tests - Test individual components in isolation
  • Integration Tests - Test interactions between components
  • Functional Tests - Test complete features or user stories
  • Performance Tests - Test system performance under load
  • Security Tests - Identify security vulnerabilities
  • Acceptance Tests - Verify the system meets business requirements

Testing Best Practices in Pipelines

  • Run fast tests (unit tests) early in the pipeline
  • Run slower tests (integration, functional) later in the pipeline
  • Fail fast - stop the pipeline as soon as tests fail
  • Make test results visible and actionable
  • Include code coverage metrics
  • Maintain a stable test environment
Example: Jest Configuration for a JavaScript Project
// jest.config.js
module.exports = {
  // Automatically clear mock calls and instances between every test
  clearMocks: true,

  // Indicates whether the coverage information should be collected while executing the test
  collectCoverage: true,

  // The directory where Jest should output its coverage files
  coverageDirectory: "coverage",

  // Indicates which provider should be used to instrument code for coverage
  coverageProvider: "v8",

  // A list of paths to directories that Jest should use to search for files in
  roots: [
    "/src"
  ],

  // The test environment that will be used for testing
  testEnvironment: "node",

  // The glob patterns Jest uses to detect test files
  testMatch: [
    "**/__tests__/**/*.js?(x)",
    "**/?(*.)+(spec|test).js?(x)"
  ],

  // A map from regular expressions to paths to transformers
  transform: {
    "^.+\\.(js|jsx)$": "babel-jest"
  },
};

Infrastructure as Code

Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure through code instead of manual processes.

Benefits of Infrastructure as Code

  • Consistency - Infrastructure is provisioned the same way every time
  • Version Control - Infrastructure changes can be tracked and reviewed
  • Automation - Reduces manual steps and human error
  • Documentation - Code serves as documentation for the infrastructure
  • Testing - Infrastructure can be tested before deployment
  • Scalability - Easy to scale infrastructure up or down

Popular IaC Tools

  • Terraform - Cloud-agnostic infrastructure provisioning
  • AWS CloudFormation - Infrastructure provisioning for AWS
  • Azure Resource Manager - Infrastructure provisioning for Azure
  • Google Cloud Deployment Manager - Infrastructure provisioning for GCP
  • Ansible - Configuration management and orchestration
  • Chef - Configuration management
  • Puppet - Configuration management
Example: Terraform Configuration for AWS EC2 Instance
provider "aws" {
  region = "us-west-2"
}

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  tags = {
    Name = "WebServer"
    Environment = "Production"
  }
  
  vpc_security_group_ids = [aws_security_group.web.id]
  
  user_data = <<-EOF
              #!/bin/bash
              echo "Hello, World" > index.html
              nohup busybox httpd -f -p 8080 &
              EOF
}

resource "aws_security_group" "web" {
  name        = "web-server-sg"
  description = "Allow HTTP and SSH traffic"

  ingress {
    from_port   = 8080
    to_port     = 8080
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

output "public_ip" {
  value = aws_instance.web_server.public_ip
}

Real-World Example: Netflix's Deployment Pipeline

Netflix is known for its advanced deployment pipeline that enables thousands of deployments per day.

Key Components of Netflix's Pipeline

  • Spinnaker - Multi-cloud continuous delivery platform
  • Automated Canary Analysis - Automated testing of new deployments
  • Chaos Engineering - Deliberately introducing failures to test resilience
  • Blue/Green Deployments - Running two identical production environments
  • Feature Flags - Enabling or disabling features without code changes

Lessons from Netflix

  • Automate everything possible
  • Test in production (safely)
  • Design for failure
  • Use data to make deployment decisions
  • Empower teams to own their deployments

Designing Your First Deployment Pipeline

When designing your first deployment pipeline, start simple and iterate.

Steps to Design a Basic Pipeline

  1. Identify Stages - Determine the key stages your code needs to go through
  2. Define Success Criteria - What must pass at each stage?
  3. Choose Tools - Select appropriate tools for each stage
  4. Implement Automation - Automate as much as possible
  5. Monitor and Improve - Collect metrics and continuously improve

Common Pipeline Patterns

  • Basic CI/CD Pipeline - Build, test, deploy
  • Environment Promotion Pipeline - Dev, test, staging, production
  • Feature Branch Pipeline - CI for feature branches, CD for main branch
  • Parallel Testing Pipeline - Run different types of tests in parallel
  • Microservices Pipeline - Independent pipelines for each service
